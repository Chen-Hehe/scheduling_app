#### 📦 一、 全局数据结构定义 (Data Interfaces)

在开始写组件前，请基于以下数据结构理解业务逻辑：

TypeScript

```
// 用户信息
interface User {
  student_id: string; // 学号（主键/唯一标识）
  name: string;
  department: string;
  role: '部长团' | '副部长' | '干事';
}

// 班次信息
interface Shift {
  shift_id: string;
  week_type: '单周' | '双周' ;
  day_of_week: number; // 1-6 代表周一到周六
  time_slot: string; // 例如 "14:00-16:00"
}

// 志愿项
interface Preference {
  shift_id: string;
  preference_rank: number; // 1代表第一志愿，2代表第二志愿...
}
```

#### 🧑‍🎓 二、 用户端逻辑与处理细节 (C-End Flow)

**Step 1: 身份认证页 (无密码验证)**

- **UI 组件**：表单（输入框：姓名、学号）。
    
- **交互逻辑**：
    
    - 用户输入姓名和学号点击“下一步”。
        
    - **处理细节**：前端发起 API 请求校验。若后端返回匹配成功，则自动获取该用户的 `department` 和 `role` 并存入全局状态（如 Pinia/Redux），跳转至 Step 2；若失败，提示“查无此人，请核对学号或联系管理员”。
        

**Step 2: 空闲时间选择页 (核心交互)**
- **交互逻辑**：
    - **处理细节**：
        - **强制校验**：监听已选班次的数组长度 `selectedShifts.length`。当长度 `< 4` 时，底部“下一步”按钮置灰，并显示红字提示“请至少选择 4 个空闲时间段”。
            

**Step 3: 志愿排序页 (拖拽交互)**

- **UI 组件**：列表组件（支持 Drag & Drop 拖拽排序，如使用 `vuedraggable` 或 `dnd-kit`）。
    
- **交互逻辑**：
    - **处理细节**：拖拽结束后，更新数组元素的 `preference_rank` 属性。点击“提交”按钮，将 `student_id` 和排序后的数组整体 POST 给后端。
        

#### 👨‍💼 三、 管理员端逻辑与处理细节 (B-End Flow)

**模块 1: 花名册与填报进度管理**

- **UI 组件**：两个 Tab（“已提交”、“未提交”）、数据表格（Table）、文件上传组件。
    
- **交互逻辑 & 处理细节**：
    
    - **名单导入**：提供“导入花名册(Excel)”按钮。前端读取 Excel / CSV 文件并解析为 JSON 数组传给后端，完成系统初始化。
        
    - **未交催办**：在“未提交”Tab 下，提供一个“一键复制未交名单”按钮。点击后，遍历未提交表格的数据，将姓名拼接成 `@张三 @李四` 格式并写入系统剪贴板，提示“复制成功，请前往群聊粘贴”。

**模块 3: 智能排班与结果微调 (复杂交互)**

- **UI 组件**：带人员标签的可视化排班表格（类似日历排班表）。
    
- **交互逻辑 & 处理细节**：
    
    - **一键排班**：点击按钮，触发后端排班算法，并 loading 等待返回排班结果。
        
    - **结果渲染**：将后端返回的 `Schedules` 数据渲染到表格中。每个班次单元格内显示被安排的人员姓名和角色（可用不同颜色 Tag 区分部长和干事）。
        
    - **手动微调（重要）**：如果某单元格人数不够，或者需要换人，管理员点击该单元格/人员标签，弹出一个抽屉(Drawer)或弹窗(Modal)。弹窗内列出“该时段有空闲意向的其他人员名单”，点击替换即可更新排班状态。
        
    - **导出结果**：提供“导出排班表”按钮，前端利用 `xlsx` 库或直接请求后端文件流，将最终表格下载为 Excel 文件。
